# 백엔드 마이크로서비스 설계 및 전환 계획 (Backend MSA Design & Migration Plan)

본 문서는 레거시 시스템을 코틀린(Kotlin) 및 최신 스프링 부트(Spring Boot) 기반의 마이크로서비스로 전환하기 위한 구체적인 아키텍처 설계와 모듈 분리 전략을 정의합니다.

## 1. 기술 스택 선정 및 버저닝 전략

### 1.1 언어 및 프레임워크
- 언어: Kotlin (코루틴을 활용한 비동기 처리 및 Null 안정성 확보)
- 프레임워크: Spring Boot 3.4.x 최신 안정화 버전 + Spring Cloud (2023.0.x 이상)
- 가상 스레드 (Virtual Threads): 입출력 바운드 작업의 효율성을 위해 자바 21의 가상 스레드를 기본으로 사용합니다.

### 1.2 메모리 최적화 전략
- t4g.nano/micro 환경을 고려하여 마이크로서비스별 최대 힙 메모리(-Xmx)를 엄격히 제한합니다.
- 상세 내용은 docs/spring-boot-memory-optimization-guide.md를 준수합니다.

## 2. 인증 및 권한 관리 전략

MSA 환경에서 보안의 복잡성을 줄이고 각 서비스의 독립성을 보장하기 위해 다음과 같은 전략을 사용합니다.

### 2.1 게이트웨이 기반 JWT 검증 및 헤더 삽입
- 모든 클라이언트 요청은 게이트웨이(microservice-gateway)를 통해 진입하며, Authorization 헤더에 JWT를 포함합니다.
- 게이트웨이는 JWT 토큰의 유효성을 중앙에서 검증하고, 페이로드에서 사용자 식별 정보(User ID, Role 등)를 추출합니다.
- 추출된 정보를 X-User-Id, X-User-Role 등의 커스텀 HTTP 헤더에 삽입하여 내부 마이크로서비스로 전달합니다.
- 내부 마이크로서비스는 게이트웨이가 전달한 헤더 정보를 전적으로 신뢰하며 중복 검증을 수행하지 않습니다.

## 3. 마이크로서비스 구성 및 역할

도메인별 응집도와 배포의 독립성을 고려하여 다음과 같이 서비스를 구성하며, 각 서비스는 독립된 Git 서브모듈 저장소로 관리합니다.

### 3.1 서비스별 책임
- microservice-user: 회원 가입, 인증/인가(OAuth), 사용자 프로필 및 권한 관리
- microservice-book: 책 정보 제공, 검색, 큐레이션 및 메타데이터 관리
- microservice-talk: 사용자 간 대화, 피드백(반응) 등 실시간 커뮤니케이션 처리
- microservice-customer: 고객 센터 문의 접수 및 CS 업무 처리
- microservice-gateway: 모든 요청의 단일 진입점으로 라우팅 및 공통 보안 처리

### 3.2 서비스 탐색 및 설정 관리 (Consul)
- 서비스 탐색: HashiCorp Consul을 활용하여 모든 마이크로서비스의 위치를 동적으로 관리합니다.
- 설정 관리: 모든 서비스의 환경 설정값은 Consul의 Key-Value 저장소에서 중앙 집중식으로 관리하며 실시간 업데이트를 지원합니다.
- 헬스 체크: 인프라 수준에서 서비스의 생존 여부를 주기적으로 확인하여 게이트웨이 라우팅에 반영합니다.

## 4. 향후 작업 순서 (Action Items)

- 기반 서비스 포팅: 기존 레거시의 회원 가입 및 인증 로직을 microservice-user로 마이그레이션합니다.
- 게이트웨이 라우팅 설정: Consul과 연동하여 각 서비스별 라우팅 규칙을 정의합니다.
- 데이터베이스 분리: 각 마이크로서비스가 자신만의 독립적인 데이터베이스 스키마를 갖도록 전환합니다.
- 모니터링 구축: Prometheus와 Grafana를 연동하여 각 서비스의 리소스 사용량 및 지표를 가시화합니다.

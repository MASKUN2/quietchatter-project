# 스프링 부트 메모리 최적화 가이드

메모리가 제한적인 인프라 환경에서 마이크로서비스의 성능과 안정성을 확보하기 위한 애플리케이션 레벨의 최적화 지침입니다. JVM 튜닝 외에 스프링 부트 및 애플리케이션 설정에서 준수해야 할 사항들을 정의합니다.

## 1. 가상 스레드 적극 활용

- 자바 21 이상 및 스프링 부트 3.2 이상 환경에서 가상 스레드를 기본으로 사용합니다.
- application.yml 파일에 spring.threads.virtual.enabled: true 옵션을 명시적으로 추가합니다.
- 기존 플랫폼 스레드가 운영체제 메모리를 크게 점유하는 반면 가상 스레드는 힙 메모리에 적은 용량으로 생성됩니다. 다량의 네트워크 호출이나 데이터베이스 접근과 같은 입출력 작업 시 메모리 고갈과 컨텍스트 스위칭 비용을 최소화할 수 있습니다.

## 2. 불필요한 의존성 및 자동 설정 최소화

- build.gradle.kts 파일에서 실제로 사용하지 않는 스프링 부트 스타터 라이브러리 및 외부 의존성을 정기적으로 검토하여 제거합니다. 이를 통해 불필요한 클래스가 메모리에 로드되는 것을 방지합니다.
- 마이크로서비스의 역할에 맞지 않는 자동 설정은 SpringBootApplication 어노테이션의 exclude 속성을 통해 명시적으로 비활성화합니다. 예를 들어 데이터베이스를 사용하지 않는 서비스에서 관련 자동 설정이 동작하여 자원을 소모하는 것을 차단해야 합니다.

## 3. 톰캣 내장 웹 서버 연결 제한 튜닝

- 가상 스레드를 사용하더라도 시스템이 동시에 감당할 수 있는 최대 연결 수는 인프라의 가용 메모리 용량에 비례하여 제한해야 합니다.
- application.yml 파일에서 server.tomcat.max-connections와 server.tomcat.accept-count 값을 보수적으로 설정합니다. 처리 한계를 초과하는 트래픽이 유입될 때 요청이 무한정 큐에 쌓여 메모리가 폭증하고 시스템이 멈추는 현상을 방어합니다.
- 각 마이크로서비스의 특성과 할당된 메모리 크기에 맞춰 부하 테스트를 진행하고 최적의 임계치를 산출하여 적용해야 합니다.

## 4. 로깅 수준 및 방식 최적화

- 로깅 과정에서의 잦은 문자열 결합 연산은 수명이 짧은 객체를 대량으로 생성하여 가비지 컬렉터에 부담을 주고 메모리 사용량을 급증시킵니다.
- 운영 환경의 기본 로그 레벨은 정보(INFO) 또는 경고(WARN) 수준으로 상향 고정하여 상세한 디버그 로그가 불필요하게 메모리를 소모하지 않도록 통제합니다.
- 코드를 작성할 때는 파라미터화된 로깅 방식(예: log.info("데이터: {}", data))을 엄격히 준수하여 로그가 출력되지 않는 상황에서도 문자열 객체가 생성되는 것을 원천적으로 차단해야 합니다.
